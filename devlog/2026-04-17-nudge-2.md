  - iOS `NudgeSync.swift` `pushLocalChange` 에 **`iPhone:push skipped/start/OK/FAIL`** appendDebugLog 추가 (iOS 쪽도 outbound 가시화 없었음)
  - 세 파일의 헤더 코멘트를 Complication 컨벤션에 맞춰 "3개 타겟 복제" 안내문으로 통일, 각 파일이 "← 이 파일" 표시

## 🤔 결정 사항

- **정합화 범위를 "Watch recv 3줄"에서 "양쪽 push 로그 추가"까지 확장.** 원래는 Watch 가 뒤쳐진 recv 로그만 메우면 됐지만, iOS push 측에도 appendDebugLog 가 없어 iPhone→Watch outbound 가 🐜 에 안 찍힘. 둘 중 하나만 고치면 절반만 보여서 또 한 번 라운드트립이 필요해짐. 대칭으로 한 번에 정리.
- **Complication NudgeSync 는 건드리지 않음.** 이미 `pushAwaitingActivation` 에 `pushAwait:*` 로그가 풍부. 관례가 다른(push-only) 컨텍스트이므로 iOS/Watch 의 in-process push 로그 컨벤션과 별개로 유지.
- **드리프트를 영구적으로 막을 방법은 별도 작업으로 미룸.** 옵션: (a) `Sources/` 공용 폴더 + pbxproj 에서 세 타겟이 같은 파일을 참조, (b) Swift Package 로 빼서 3 타겟이 의존. 지금은 주석으로 "3곳 동시 수정 필요" 경고만 둠. 버그 해결 후 정리하는 게 합리적.

## 🐛 문제와 해결 과정

- **문제:** 현태님이 "저번에 로그 들여다봤는데 내용 기억 안 남" 이라고 함. 로그가 🐜/🐞 에 찍히긴 찍혔던 것. 하지만 드리프트 때문에 Watch 쪽 `didReceiveUserInfo` 는 `iPhone:recv` 로 전혀 찍힌 적이 없음(iOS 파일에만 있고 Watch 파일엔 없던 라인). 즉 iPhone→Watch 방향이 조용히 숨겨져 있었음.
- **해결:** 실기기 재현 전에 로그 대칭화부터. 재현 후 로그 패턴으로 원인 격리 가능:
  - Watch 컴플리케이션 탭 → iPhone 🐜:
    - `pushAwait:entry` 없음 → Intent 자체가 실행 안 됨 (bundle id/entitlements/시계 화면 배치 문제)
    - `pushAwait:TIMEOUT` → Widget Extension 수명보다 WCSession 활성화가 더 오래 걸림 (아키텍처 피벗 필요)
    - `pushAwait:updateApplicationContext OK/transferUserInfo queued` 는 찍히는데 iPhone 에 `iPhone:recv ...` 가 없음 → iOS 측 delivery 지연/드롭 (iPhone 백그라운드 상태, WCSession 비활성화)
    - iPhone `iPhone:recv applicationContext` 까지 찍히는데 UI 미반영 → `applyRemoteSnapshot` 의 `lastModified` 비교에서 equal-or-older 로 버려지거나 `nudgeDataChangedRemote` NotificationCenter 경로 파손
  - iPhone +1 → Watch 🐞:
    - `iPhone:push start paired=X installed=Y reachable=Z` 없음 → `pushLocalChange` 호출 자체 안 됨 (ContentView 에서 +1 → `pushLocalChange()` 경로 깨짐 / 디바운스)
    - `iPhone:push skipped state=0` → WCSession 활성화 실패
    - `iPhone:push OK updateApplicationContext mod=N` 은 찍히는데 Watch 에 `Watch:recv ...` 없음 → Watch 앱 포그라운드 여부, 컴플리케이션은 recv 안 한다는 점(Watch 앱이 실행 중이어야만 applicationContext 배송) 가능성
    - `Watch:recv applicationContext` 까지 찍히는데 컴플리케이션 숫자 안 바뀜 → `WidgetCenter.shared.reloadAllTimelines()` 가 iOS 조건부 블록에 막혀 있음(현 코드는 `#if os(iOS)` 안쪽에 있음) — **이건 드리프트 정합화하면서 발견한 잠재 버그**. Watch 쪽 `handleRemote` 에선 timeline reload 가 호출되지 않음. 실기기 로그로 확증되면 `#if os(watchOS)` 분기에서도 `WidgetCenter.reloadAllTimelines()` 호출 추가 필요.

- **교훈:** "로그는 있었다"와 "로그가 완전하다"는 다르다. 드리프트가 진단용 로그 일부를 조용히 숨기고 있으면 디버깅이 공회전함. 버그 고치기 전 진단 인프라부터 대칭화.

## 📅 다음에 할 일 (실기기 세션 스크립트)

### A. Warm-up (앱 세션 활성화)
1. iPhone 에서 Nudge 앱 실행 → 5초 체류 → 🐜 클릭 → **"Clear"** 로 debugLog 리셋 → 앱 나가지 말기
2. Watch 에서 Nudge 앱 실행 → 5초 체류 → "🐞 Debug" 들어가서 **"Clear"** → 손목 내려서 앱 빠짐 허용
3. 시계 화면 꾸미기에 Nudge 컴플리케이션 3종(Circular/Inline/Rectangular) 중 **Circular** 최소 하나 배치

### B. 방향 1 (Watch Complication → iPhone)
4. iPhone 앱 백그라운드(홈 버튼)
5. Watch 시계 화면에서 **Nudge Circular 컴플리케이션 탭 1회** → 햅틱/숫자 반응 관찰
6. iPhone 앱으로 돌아와서 🐜 로그 스크롤 → 아래 3개 패턴 중 어느 것이 보이는지 기록:
   - `pushAwait:entry` → `pushAwait:state=...` → `pushAwait:updateApplicationContext OK mod=...` → (iPhone 쪽) `iPhone:recv applicationContext` → 숫자 반영 ✓
   - `pushAwait:entry` → ... → `pushAwait:TIMEOUT` → 끝. iPhone 엔 아무 recv 없음 ✗
   - `pushAwait:entry` 조차 없음 → Intent 자체 실행 안 됨 ✗

### C. 방향 2 (iPhone → Watch)
7. iPhone 앱에서 활성 운동 선택 후 **+1 버튼** 누르기 1회
8. 바로 iPhone 🐜 로그 확인 → `iPhone:push start paired=? installed=? reachable=?` → `iPhone:push OK updateApplicationContext mod=...` 가 찍혔는지
9. Watch 앱 다시 열기 → 🐞 Debug → `Watch:recv applicationContext` 가 찍혔는지, iPhone push 시각 이후인지
10. Watch 컴플리케이션 숫자가 바뀌었는지 확인. Watch 앱 카운터 숫자도 확인.

### D. 로그 수집
11. 🐜 로그 스크린샷 or 텍스트 복사 (iOS DebugLogView 는 Clear/Refresh 있지만 복사 버튼 없으면 스크린샷으로)
12. 🐞 Debug 로그 스크린샷
13. 두 로그 타임라인을 비교해서 어느 스테이지에서 끊기는지 기록

### E. 예상 다음 패치 후보 (로그 패턴별)
- 패턴 "Watch push OK → iPhone recv 없음" → iPhone 앱 백그라운드 시 WCSession delivery 지연. 대책: iPhone 이 포어그라운드 복귀 시 pull 로직 추가 / `transferUserInfo` 만으로 queue 배송 기대
- 패턴 "iPhone push OK → Watch recv 없음" → Watch 앱 포어그라운드 조건. 대책: 컴플리케이션 쪽에서도 WCSessionDelegate 로 recv 하도록 타겟 확장 (현재 컴플리케이션 타겟에는 recv 없음 — push-only 구조)
- 패턴 "Watch recv OK → 컴플리케이션 숫자 미반영" → **현재 `handleRemote` 의 `WidgetCenter.reloadAllTimelines()` 호출이 `#if os(iOS)` 안에만 있음**. watchOS 분기에서도 호출 추가 필요. (정합화 과정에서 발견한 잠재 버그)

## 💭 느낀 점

- 핵심 파일이 3 타겟에 수동 복제되는 구조는 코드량은 작아도 리스크가 크다. 한쪽만 수정하면 진단 로그가 조용히 비대칭이 되고, 그 비대칭은 "감춰진 정보"로 작용해서 디버깅이 공회전한다. 실제로 이번에 `iPhone:recv` 라인이 Watch 파일에 없던 탓에 현태님 기억 속 로그가 절반짜리였을 가능성이 높음.
- Complication 의 `pushAwaitingActivation` 은 위젯 프로세스 수명 안에서 WCSession 활성화를 기다리는 흥미로운 패턴이지만 5초 타임아웃이 실기기에서 충분한지는 검증된 바 없음. 로그 보면서 실측해야 함. 만약 자주 TIMEOUT 이 찍히면 `transferUserInfo` 만 써서 "queue 에 올려놓고 세션이 나중에 배송" 방식으로 단순화하는 선택지도 있음.
- "정합화 세션 + 진단 로그 보강 → 실기기 재현" 순서가 맞았다. 바로 실기기로 갔으면 Watch 쪽 로그가 비어 있어서 똑같이 "양방향 다 안 된다"만 재확인하고 돌아왔을 것.
