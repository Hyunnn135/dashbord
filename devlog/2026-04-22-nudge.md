## 📌 오늘 한 일

**지난 커밋의 전제 오류 발견.** 04-17 커밋 `9528a87` ("fix(sync): remove #if os(iOS) gate around WidgetCenter.reloadAllTimelines in handleRemote") 의 커밋 메시지가 "complication copy has no handleRemote so untouched" 라고 단정하고 컴플리케이션 타깃을 건드리지 않았음. 오늘 실기기 테스트 들어가기 전에 코드 한 번 더 훑어보다가 `NudgeWatchComplication/NudgeSync.swift:180-196` 에 `handleRemote` 가 **존재**한다는 걸 확인. 04-17 세션에서 그 파일을 read-only 로만 본 게 그대로 정합화 누락으로 이어짐.

**Complication NudgeSync.swift 정합화 (A/B/C).** iOS/Watch 와 동일한 구조로 맞춤:

- **A. `import WidgetKit` 가드 밖으로 이동** — `#if os(iOS) ... import WidgetKit ... #endif` 구조라 watchOS 빌드에서는 WidgetKit 심볼이 아예 보이지 않았음. 가드 밖으로 빼서 iOS 14+/watchOS 9+ 양쪽에서 사용.
- **A. `handleRemote` 내부 `#if os(iOS)` 가드 제거** — `WidgetCenter.shared.reloadAllTimelines()` 이 `#if os(iOS) ... #endif` 에 갇혀 있었음. watchOS 에서 수신 경로가 드물긴 하지만, `transferUserInfo` 는 큐잉되므로 컴플리케이션이 다음 실행 시 pending userInfo 를 수신 → handleRemote 실행 → reload 안 되는 구조적 버그 가능성이 존재. 가드 제거 + `handleRemote:reloadAllTimelines changed=true` appendDebugLog 추가.
- **B. `didReceiveUserInfo(_:)` 델리게이트 추가** — iOS/Watch 에는 있는데 컴플리케이션만 없었음. `transferUserInfo` 가 `updateApplicationContext` 와 달리 영속 큐 기반이라 userInfo 경로 수신 가능성 존재. 구현은 iOS/Watch 와 동일 시그니처 + `Comp:recv userInfo` appendDebugLog.
- **C. recv 3종 + pushLocalChange 에 `Comp:` 접두사 appendDebugLog** — iOS/Watch 는 `iPhone:` / `Watch:` 로 정리돼 있지만 컴플리케이션은 `print` 만 있고 🐞 디버그 뷰어에서는 증발 상태. `Comp:push skipped/start/OK/FAIL` + `Comp:recv applicationContext/message/userInfo` 로 보충. 기존 `pushAwait:*` 로그는 그대로(이건 컴플리케이션 고유 경로).

**Watch ContentView 에 `WKApplication.didBecomeActiveNotification` 훅 추가 (D).** iOS `ContentView` 는:

```swift
.onReceive(NotificationCenter.default.publisher(for: UIApplication.didBecomeActiveNotification)) { _ in
    refresh()
    NudgeSync.shared.pushLocalChange()
}
```

라는 훅으로 위젯 탭 후 앱 복귀 시 최신값 반영 + iPhone→Watch push 를 보장하는데 Watch `ContentView` 에는 대응 훅이 없었음. 컴플리케이션 탭 → 카운트 증가 → Watch App 실행 → 포그라운드 복귀 시 Watch 쪽에서 iPhone 으로 **강제 push 가 누락**되는 2차 용의자. watchOS 26.4 대응 `WKApplication.didBecomeActiveNotification` 기반 `onReceive` 추가. 기존 `.nudgeDataChangedRemote` 훅은 유지.

**3-copy diff 검증.** 정합화 후 각 파일에서:
- `import WidgetKit` 라인 위치: 세 파일 모두 line 26 (`#if os(iOS)` 블록 밖)
- `handleRemote` 내 `#if os(iOS)` 가드: 0개 (FAIL 없음)
- `WidgetCenter.shared.reloadAllTimelines()` 호출: 1개씩
- `didReceiveUserInfo` 델리게이트: 세 파일 모두 존재
- `handleRemote:reloadAllTimelines changed=true` debug log: 세 파일 모두 존재

iOS vs Watch `diff` 결과는 `iPhone:` ↔ `Watch:` 접두사 차이 3줄만. Complication 은 `pushAwaitingActivation` 메서드(약 40줄)가 추가로 있고 접두사가 `Comp:` 이라는 점만 의도적 분기.

## 🤔 결정 사항

- **수신 경로 실질적 의미는 낮지만 정합화는 필수.** 컴플리케이션 extension 은 단명(1~2초) 프로세스라 실제로 `applicationContext` 수신할 가능성은 Watch App 대비 낮음. 하지만 (a) 3-copy 드리프트 방지 규칙상 의도적 분기를 제외하고 같아야 유지보수 가능하고, (b) `transferUserInfo` 는 큐잉되므로 Complication 이 탭 순간 pending userInfo 를 받을 수 있음 → 이 경우 `reloadAllTimelines` 가 `#if os(iOS)` 로 막혀 있으면 컴플리케이션 타임라인 stale. 실제 영향 가능성 낮지만 증거 있는 버그는 고치는 쪽으로.
- **실기기 가기 전에 코드 정리부터.** Option A(먼저 코드 정리 → 실기기) vs Option B(실기기 먼저 로그 → 증거 기반 수정) 중 A 선택. 이유: 지금 상태는 `9528a87` 이 iOS/Watch 만 고치고 Complication 만 드리프트된 채 남아있는 중간 상태. 실기기 로그 받아도 "이 로그가 드리프트 때문인지 버그 때문인지" 먼저 구분해야 하는 비용이 오히려 커짐. 정합화는 10분 작업이고 빌드만 확인하면 끝.
- **Watch 접두사는 기존 그대로 유지(`Watch:`), 컴플리케이션은 `Comp:`.** watchOS 상에서 두 프로세스가 로그를 공유하지만(App Group 동일) 서로 다른 바이너리이므로 로그 분석 시 구분자 필수. 기존 04-17 작업에서 `Watch:` 는 이미 박혀있어 그대로 두고 컴플리케이션만 신규로 `Comp:` 추가.

## 🐛 문제와 해결 과정

- **문제:** 04-17 커밋 메시지가 "complication copy has no handleRemote so untouched" 로 단정 → 사실과 다름. 컴플리케이션은 iOS/Watch 와 달리 `#if canImport(WatchConnectivity)` 블록 아래에 WCSessionDelegate extension 을 완전히 구현하고 있고 `handleRemote` 가 180-196 라인에 **존재**. 지난 세션이 이 파일을 전수 읽지 않고 추정으로 결정.
- **해결:** 오늘 실기기 가기 전 모든 3-copy 를 `wc -l` + `grep` + `diff` 로 교차 검증. 검증 체크리스트:
  - `import WidgetKit` 위치
  - `handleRemote` 안에 `#if os(iOS)` 가드 여부
  - `WidgetCenter.shared.reloadAllTimelines()` 호출 수
  - `didReceiveUserInfo` 존재 여부
  - `handleRemote:reloadAllTimelines` appendDebugLog 존재 여부
  
  각 항목을 스크립트로 세 파일에서 동일하게 체크해서 어긋난 곳 포착. 이 체크를 다음번 "3-copy 가 같은지" 검증 시에도 재사용 가능.
- **교훈:** 커밋 메시지에 적힌 전제도 정합화 체크 대상이다. "handleRemote 없음" 같은 단정은 `grep` 1초로 사실 확인 가능한데, 그걸 건너뛰면 다음 세션에서 "지난 세션이 놓친 버그" 를 반복해서 발견하게 됨. 드리프트 방지 규칙을 "파일이 같아야 한다" 에서 "파일이 같은지 매 커밋 전에 교차검증한다" 로 수준 업 필요.
