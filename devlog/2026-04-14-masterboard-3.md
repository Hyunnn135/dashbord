## 📌 오늘 한 일

홈 대시보드의 작은 불편들과 구조적 불일치를 한 번에 정리. 사용자 체감상 "홈이 드디어 내 것처럼 느껴진다"가 목표.

### 1. 네비바 날짜 표시 제거
- `index.html` 네비바 우측의 `nav-date` 요소와 그 값을 주입하던 스크립트 제거.
- `project-masterboard.html`도 같은 자리에 `last-updated` 배지가 있어 함께 제거하고 스크립트 할당도 no-op 처리 (다른 보드는 원래 없었음).
- 이유: 4개 보드가 네비 디자인만은 완전 동일해야 "어느 페이지든 같은 앱"이라는 일관성이 유지되는데, 홈만 날짜 배지를 달고 있어 오히려 어색했음.

### 2. 프로젝트 카드 드래그&드롭 + 자동정렬
- 홈의 `.project-grid`에 HTML5 Drag-and-Drop API 적용.
- 각 카드에 `draggable="true"` + `data-card-id`(salarykorea / realestate / maxout / nudge) 부여.
- `dragstart → dragover(위치 계산) → drop` 흐름으로 DOM 재배치. 커서 위치 기준으로 "왼쪽(위)" vs "오른쪽(아래)" 판정해 자연스러운 삽입 지점 결정.
- 드래그 직후 클릭 이벤트(`<a>` 카드의 페이지 이동)를 억제하는 `dragSuppressClick` 플래그 추가 — 놓자마자 실수로 페이지 이동하는 것 방지.
- 변경된 순서는 `localStorage['mb_card_order_v1']`에 배열로 저장, 재방문 시 자동 복원.
- **자동정렬 버튼**(`↻ 자동정렬`): localStorage 초기화 후 reload → HTML 선언 순서(기본값) 복귀. 사용자가 언제든 원상복구 가능.
- 카드 커서를 `grab`/`grabbing`으로 변경, 드래그 중 `.dragging`(투명도↓) + 호버 대상에 `.drag-over`(테두리 하이라이트) 시각 피드백 추가.

## 🤔 결정 사항

- **드래그&드롭 구현은 네이티브 HTML5 API만 사용** — SortableJS 같은 라이브러리를 도입할 필요까진 없다고 판단. 카드 4장, 한 페이지 한정 기능이라 외부 의존성 추가는 오버킬. 구현량도 ~80줄 수준.
- **순서 저장소로 localStorage 선택** — GitHub Pages 정적 사이트라 서버 저장은 불가하고, 이 기능은 "이 브라우저에서 내가 보고 싶은 순서"라서 디바이스별 로컬 저장이 오히려 자연스러움. 폰에서 보는 순서와 맥에서 보는 순서가 달라도 OK.
- **자동정렬 = 기본 순서로 되돌리기** — "알파벳순" 같은 규칙 기반 정렬을 넣을까 고민했지만, 프로젝트 중요도는 HTML 선언 순서에 이미 담겨 있음(가장 활발한 salarykorea가 맨 앞). 그래서 "수동 정렬 해제"를 자동정렬로 정의.
- **devlog 렌더링은 클라이언트 사이드 marked.js** — GitHub Pages 특성상 서버 렌더링 불가. pre-build로 HTML 생성하는 안도 있었지만, "devlog 하나 추가할 때마다 빌드 스텝 돌리기"는 WORKFLOW_RULES의 "4곳 동시 업데이트" 부담을 늘릴 뿐. 매니페스트 한 줄 추가만으로 끝나게 설계.
- **블로그 카테고리는 월급연구소로 흡수하되 업로드 툴은 유지** — 네이버 블로그 65건 업로드 작업 자체는 물리적 도구가 필요하므로 `blog-upload-board.html`은 존속. 개념상 하위 도구로 네비 링크만 남김.
- **프로젝트 명칭 포맷 "한글 (영어)"** — 한국어 사용자가 제1 독자이고, 영어명은 폴더명/식별자 역할이라 괄호로 부차 정보임을 표시. 포트폴리오 맥락에서 영어권 독자가 볼 때도 두 표기가 모두 있어 유리.

## 🐛 문제와 해결 과정

- **`<a>` 카드에 draggable을 붙이면 기본 동작이 이미지 URL을 이미지로 드래그하는 것이 돼서 이상함** — 해결: 카드 내부에 이미지가 없어 이 문제는 발생 안 함. 대신 드래그 후 클릭으로 페이지 이동되는 이슈는 `dragSuppressClick` 플래그로 직접 차단.
- **드래그 대상을 "오른쪽에 놓기"인지 "왼쪽에 놓기"인지 판정** — 초기에는 인덱스 비교만 썼더니 카드가 작은 그리드(넓이 > 높이*1.2)일 땐 좌우 기준, 세로로 긴 모바일에선 상하 기준이어야 자연스러움. `isWide` 분기 추가.
- **marked.js CDN이 드물게 차단되는 환경** — fetch catch에서 "index.json 로드 실패" 메시지를 표시하고, 사이드 목록에서 항목 클릭 시에도 `원문 보기 (raw)` 링크로 fallback 가능하게 함.
- **section-label의 margin-bottom이 `section-head` 내부에 넣으면 어긋남** — 추가 규칙 `.section-head .section-label{margin-bottom:0}`로 해결. 타임라인 섹션처럼 단독 사용되는 경우는 기존 margin 유지.

## 💭 느낀 점

- "홈이 드디어 내 것처럼 느껴진다"가 체감됨. 5개 요청 중 4번(이름 통일)과 5번(devlog 뷰어)이 가장 큰 변화였지만, 1번(날짜 제거)이 **시각적 안정감**을 만든 일등공신. 작은 디테일이 의외로 큰 인상 차이를 만든다는 걸 다시 확인.
- devlog 뷰어를 만들면서 "그동안 쓴 글이 읽히는 경험"의 가치를 실감. 파일로만 쌓아두니 돌아볼 동기가 없었는데, 브라우저에서 목록으로 보이니 어제/그저께 기록이 살아있는 포트폴리오로 느껴짐.
- 드래그&드롭 같은 "아주 흔한 UX"도 막상 구현하면 `dragSuppressClick`, `<a>` 요소 기본 동작 방지, 가로/세로 그리드 판정 같은 엣지 케이스가 수두룩함. 라이브러리 쓰는 것이 왜 일반적인지 체감했지만, 4장짜리 그리드에 60KB 라이브러리는 여전히 과함.
- "블로그 = 월급연구소 서브 카테고리"라는 인식 전환이 개념적으로 가장 중요한 변경. 앞으로 신규 사이트 블로그가 생기면 그 프로젝트 카드의 서브로 묶어야 한다는 원칙 확립.
- 이 3차 세션 포함해서 하루에 devlog 3편 작성 — 마스터보드 + nudge + 마스터보드-2 + 마스터보드-3. WORKFLOW_RULES가 없었으면 분명 "3차 세션은 나중에 정리하자"며 미뤘을 일. 규칙이 관성을 만드는 중.
