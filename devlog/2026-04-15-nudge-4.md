Phase 3 는 조합 경우의 수가 많아서 작업 전에 방향성 3개 먼저 확정:

| 질문 | 선택 |

CloudKit 은 schema 설계 / subscription / conflict 처리로 복잡도가 확 올라가는 데 비해, Nudge 페르소나(운동 중 바로 카운트) 에는 WC 의 peer-to-peer 즉시성이 맞음. iCloud 계정 의존도도 낮출 수 있음. 단점은 watch 단독 셀룰러 상태에서 iPhone 완전 꺼져 있으면 미루어진다는 점인데, 이건 실사용 빈도 ≈ 0 으로 판단.

### 2) Watch 타겟 pbxproj 복원

2차 세션에서 Watch 타겟을 임시 비활성화 할 때 백업해둔 `project.pbxproj.backup-pre-watch-disable` 와 현재 파일 diff 를 떠서 사라진 5 개 섹션 확인:

- nudge iOS 타겟의 `buildPhases` 에 Embed Watch Content 항목 추가

백업으로 통째 되돌리면 3차 세션에서 추가한 NudgeWidget entitlements 설정이 날아가서, **5 개 섹션만 정밀하게 재삽입** 하는 방식으로 복원. 복원 후 `diff backup current` 찍어서 정확히 widget entitlement 추가분 2 줄만 남는 걸 확인.

### 3) Watch 앱 entitlements

`NudgeWatch Watch App/NudgeWatch.entitlements` 파일 새로 만들고 `group.site.salarykorea.nudge` (iOS·위젯과 동일 App Group) 추가. ### 4) SharedStore 에 동기화 기반 추가

last-writer-wins 를 위해 타임스탬프 도입:

```swift
private static let lastModifiedKey = "lastModified"

static var lastModified: TimeInterval { ... }
static func touch(_ date: Date = Date()) { lastModified = date.timeIntervalSince1970 }

static func syncSnapshot(daysBack: Int = 60) -> [String: Any]
static func applyRemoteSnapshot(_ payload: [String: Any]) -> Bool
```

- `activeExercise` setter, `increment`, `decrement` 에서 `touch()` 호출
- `syncSnapshot` 은 최근 60 일 치만 포함 (updateApplicationContext 는 4KB 제한, 오래된 기록은 각 기기에 로컬 보존)
- `applyRemoteSnapshot` 은 `remote.lastModified > local.lastModified` 일 때만 덮어씀. 원격이 보내지 않은 오래된 날짜는 로컬 그대로 유지 (merge). 값 변경 있으면 `true` 반환

### 5) NudgeSync — WatchConnectivity 레이어

`nudge/NudgeSync.swift` + `NudgeWatch Watch App/NudgeSync.swift` (동일 코드 복제 — 상단 주석에 "양쪽 같이 수정" 명시).

핵심:
```swift
final class NudgeSync: NSObject, ObservableObject, WCSessionDelegate {
    static let shared = NudgeSync()

    func activate() { WCSession.default.delegate = self; WCSession.default.activate() }
    func pushLocalChange() {
        try? WCSession.default.updateApplicationContext(SharedStore.syncSnapshot())
    }
    func session(_:didReceiveApplicationContext:) { handleRemote(...) }

    private func handleRemote(_ payload: [String: Any]) {
        if SharedStore.applyRemoteSnapshot(payload) {
            NotificationCenter.default.post(name: .nudgeDataChangedRemote, object: nil)
            #if os(iOS)
            WidgetCenter.shared.reloadAllTimelines()
            #endif
        }
    }
}
```

- `#if canImport(WatchConnectivity)` 로 iOS/watchOS 공통 동작, `#if os(iOS)` 로 iOS 전용 콜백(sessionDidBecomeInactive/Deactivate, WidgetCenter) 분기
- 전송 방식은 `updateApplicationContext` — 최신 값만 보장 배달 (중간 값 drop 허용). Nudge 는 "현재 카운트 스냅샷"이 의미라 딱 맞음
- `sendMessage` 는 쓰지 않음 (실시간이지만 피어 가동 필요 → v1 범위 밖)

### 6) iOS 측 hook 연결

- `ContentView`: +1 / −1 / 세그먼트 전환 모두 `NudgeSync.shared.pushLocalChange()` 추가
- `onReceive(didBecomeActiveNotification)` 에도 push 호출 — **위젯 extension 에서 찍힌 변경분** 을 메인 앱이 foreground 되는 시점에 Watch 로 밀어줌 (위젯 extension 에서 직접 WC 보내는 건 라이프사이클 불안정해서 의도적으로 회피)
- `nudgeApp` 에 `@StateObject private var sync = NudgeSync.shared` 로 앱 시작 시점에 WCSession.activate 강제
- `.nudgeDataChangedRemote` notification 구독 → `refresh()` 로 UI 갱신

### 7) Watch 메인 앱 재작성

Xcode 기본 템플릿("Hello, world!") 걷어내고:

```swift
Text(activeExercise.displayName)     // 상단: iPhone 이 고른 운동
Text("\(count)")                     // 중앙: 큰 숫자
Button { +1 + WKInterfaceDevice.click 햅틱 + NudgeSync.push } label: { "+1" }
```

- 활성 운동 선택 UI 없음 (iPhone 전담). `SharedStore.activeExercise` read-only 관점
- `WKInterfaceDevice.current().play(.click)` 로 촉각 피드백
- `.onReceive(.nudgeDataChangedRemote)` 로 iPhone 쪽 변경 즉시 반영

## 🤔 결정 사항

- **WatchConnectivity 단독** — CloudKit 은 Nudge 페르소나에 오버엔지니어링. 차기 기능(예: "여러 기기에서 동기화된 평생 누적") 이 추가되면 그때 CloudKit 검토.
- **Last-writer-wins 타임스탬프** — 두 기기가 동시에 disconnected 상태로 카운트하면 한쪽이 덮일 수 있지만, 실사용 시 "폰/워치 동시 다른 운동 카운트" 가 거의 없어 acceptable. v2 에서 per-tap delta 기반 merge 로 확장 가능.
- **활성 운동 선택은 iPhone 전담** — 워치 화면 크기 한계 + Nudge "빠르게 +1" 컨셉에 세그먼트 UI 는 noise. 워치는 "지금 뭐 하고 있는지" 수동으로 전환할 필요가 없도록, 폰에서 세팅 → 워치는 따라옴.
- **컴플리케이션은 다음 세션** — 워치용 widget extension 은 별도 타겟 추가 필요. pbxproj 에 수동으로 타겟 추가는 ~20+ 섹션 작성 필요하고 리스크 큼. Xcode UI 에서 "File → New → Target → Widget Extension (Watch)" 한 번에 처리하는 게 안전.
- **공용 코드는 또 파일 복제** — Phase 1/2 에서 SharedStore 를 두 타겟에 복제했던 것과 동일 원칙. NudgeSync 도 iOS/watchOS 두 곳에 복제 + 상단 "양쪽 같이 수정" 주석. 파일 수 5 개 넘어가면 Swift Package 로 분리 검토.

## 🐛 문제와 해결 과정

**문제 1 — 위젯 extension 에서 WCSession 활성화 불안정**
- 증상 예상: 위젯 프로세스는 짧게 살다 죽음. 그 안에서 WCSession.activate + updateApplicationContext 까지 동작 보장 어려움
- 해결: 위젯 intent(IncrementExerciseIntent) 에서는 SharedStore 만 갱신하고 WC 는 건드리지 않음. 메인 iOS 앱이 foreground 로 돌아올 때 (`didBecomeActiveNotification`) `pushLocalChange()` 를 한 번 호출해 최신 스냅샷을 밀어줌. updateApplicationContext 는 "최신 값만 배달" 이라 위젯 탭을 여러 번 했어도 한 번에 반영됨.

**문제 2 — pbxproj 선별 복원**
- 증상: 3차 세션에서 NudgeWidget entitlement 설정을 pbxproj 에 추가했는데, 2차 백업 파일로 통째 되돌리면 그 수정이 사라짐
- 해결: `diff backup current` 떠서 차이 5 개 섹션만 추출, 현재 파일에 Edit 으로 정밀 재삽입. 복원 후 역diff 찍어서 "widget entitlement 2 줄만" 차이 나는 걸 검증.

## 💭 느낀 점

Phase 3 는 "새 타겟이 얽혀서 만지기 싫다" 싶던 작업이었는데, 백업 pbxproj 덕에 복원은 순조. 2차 세션에서 "일단 Watch 껐다가 나중에 되살리자" 결정이 정확히 맞았다. Phase 1~2 안정화에만 집중하고, Phase 3 는 별건으로 분리한 게 세션 경계를 깔끔하게 만들었음.

WatchConnectivity 의 `updateApplicationContext` 가 "최신만 배달" 이라는 성질이 Nudge 데이터 모델(현재 카운트 스냅샷)에 정확히 대응돼서, 델타 전송/순서 보존 같은 분산 시스템스러운 고민이 없음. 스펙 제약이 아키텍처를 유도한 좋은 케이스. 만약 "매 탭마다 타임스탬프 delta 를 전송" 이었으면 `transferUserInfo` + 큐 관리가 필요했을 텐데, 그쪽으로 안 흐른 게 v1 스코프에 정답.

복제된 SharedStore/NudgeSync 파일이 3개/2개까지 늘어나서 Swift Package 분리 유혹이 슬슬 올라오는데, 지금 단계에선 "한 번 수정 후 md5sum 으로 확인" 루틴이 충분히 빠름. 파일 하나 더 늘어날 때 다시 판단.
